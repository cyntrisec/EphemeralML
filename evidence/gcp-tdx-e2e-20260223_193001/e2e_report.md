## Attestation Path

Confidential Space does NOT expose configfs-tsm inside the container. The attestation path used:
1. **Boot evidence:** Synthetic TDX provider (CS mode fallback — no raw MRTD/RTMR from configfs-tsm)
2. **Transport attestation:** CS Launcher JWT (fetched from `/run/container_launcher/teeserver.sock`)
3. **JWT issuer:** `https://confidentialcomputing.googleapis.com`
4. **JWT audience:** WIP provider URI (ephemeralml-pool/ephemeralml-tdx)
5. **TDX quote:** Real hardware TDX quote generated by the Launcher for the JWT

### Key Differences

1. **Boot time:** AWS Nitro enclaves start in ~15s (pre-running host). GCP CS launches a full VM + pulls container image, taking ~80s.
2. **Attestation model:** Nitro provides direct NSM device access. GCP CS abstracts attestation behind the Launcher JWT service — the container never touches raw TDX hardware.
3. **Inference speed:** GCP c3 (Sapphire Rapids) was faster at 69ms vs 90-116ms on Ice Lake. Expected given newer CPU generation.
4. **Handshake overhead:** GCP handshake includes an HTTP round-trip to the Launcher socket for JWT + TDX quote generation (~850ms). Nitro NSM is direct device access (<1ms).
